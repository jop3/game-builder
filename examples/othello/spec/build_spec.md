# Build spec — Moonstone & Obsidian (fantasy Othello)

> Fas 1 of the **spelbygge** pipeline. The brief answers WHAT/WHY; this answers
> HOW and IN WHAT ORDER, making every step checkable. Engine target is Godot 4.x
> (matching the sibling Snittet game + this repo's Godot delivery adapter).

## 0. Testability is a requirement, not an afterthought

Standard clause — never struck:

- Headless logic tests from M0. Fast (< 30 s total), deterministic, each suite in
  its own engine process with a timeout guard.
- The agent must be able to SEE the result: screenshot + record from M0
  (Xvfb + lavapipe in the cloud). The board is a static scene → a still is enough
  for most checks; the flip cascade needs a short recording.
- Deterministic mode: RNG seed via cmdline; same seed → same match (matters for
  the bot-vs-bot self-play test).
- A bot must be able to play the game from M1. Othello's machine-checkable win
  condition (brief §5) makes this trivial — if the bot *can't* play, that's a
  design bug, not a test bug.

| Decision | Choice | One-line rationale |
|---|---|---|

## 2. Architecture contract (invariants)

These let the tests survive aggressive visual iteration. Break one deliberately →
update the contract + tests in the same commit.

- **Rules are pure and headless.** `board.gd` (or `rules.gd`) is a pure data model:
  an 8×8 `int` grid (`EMPTY|MOONSTONE|OBSIDIAN`), `legal_moves(side) -> Array`,
  `apply_move(side, cell) -> flipped_cells`, `is_terminal()`, `score()`. **No
  node, no rendering, no `randf`** — RNG only in the bot, seeded. This is what
  makes the rules testable and the bot possible.
- **The 8 flip directions are one shared constant** used by both legal-move
  generation and apply — never two copies (the classic Othello bug is a direction
  table that disagrees between "is this legal" and "what flips").
- **View is a projection of the model, never the source of truth.** The scene
  reads the grid to place disc nodes; clicking a cell calls `apply_move` and then
  *animates toward the new model state*. A desync bug is impossible if the view
  never stores rules state.
- **Node names are API:** `Board`, `Cell_<r>_<c>`, `Disc_<r>_<c>` — tests and the
  bot address cells by name; renaming is a contract break.
- **Collision/pick shapes are simple and stable:** each cell is a flat box picker
  on a fixed grid; visual passes (better board mesh, disc bevels) never move the
  pickers or rename nodes → reskin-safe.
- **Generated assets, hand-tuned placement:** the board + discs come from the
  pipeline as `.glb`; the scene places 64 disc instances on a CONFIG-driven grid
  (cell size, board origin) — reproducible, one source of truth for geometry.
- **"Why" comments** on tuned constants (flip delay, arc height) so a later feel
  pass doesn't silently tune the juice back out.

## 3. Milestone staircase

Each milestone: small, observable after, with a Testable line. M0 is always the harness.

### M0 — Harness
Test runner, screenshot/record (Xvfb+lavapipe), deterministic seed, headless guards.
**Testable:** `run_tests.sh` → green < 30 s; `screenshot.sh empty out.png` → PNG with content.

### M1 — Playable slice (ugly is right: flat quads, 2 colors)
Full core loop headless-legal: start position → generate legal moves → place →
flip run → alternate turns → pass when stuck → terminal → score. Grey/white/black
quads, no juice, no generated art yet.
**Testable:** `test_rules` green (legal moves + flips correct on known positions);
the bot plays a full random-legal game to terminal and `score()` sums to 64;
a recording of one full match produced and watched.

### M2 — Graphics integration (the fantasy board + markers)
Swap the flat quads for the pipeline-generated `.glb`: board mesh, Moonstone disc,
Obsidian disc, placed on the CONFIG grid. Rules untouched (invariant check).
**Testable:** `test_rules`/`test_bot` still green (invariants held); a still of the
starting position renders with the fantasy board + 4 center discs correctly placed.

### M3 — Feel pass (the flip cascade + place-down)
The one juice pass the pitch lives on: captured discs flip one-by-one along the
captured line (delay `FLIP_STEP_S`, arc height `FLIP_ARC`, ease `FLIP_EASE`),
a soft settle bounce on placement, warm candlelit ambiance + soft shadows, disc
*clack* SFX. Nothing here touches rules/pickers/node names.
**Testable:** `test_rules`/`test_bot` still green; before/after recording of a
multi-disc capture reviewed against the feel rubric §5.

### M4 — Reference bot + endgame polish
One parameterized bot (greedy vs. corner-aware + shallow lookahead) so a match
against the AI reaches a real terminal; win/draw banner; new-match reset.
**Testable:** `test_endgame` (terminal + winner correct on crafted end positions);
bot-vs-bot self-play over N seeds always terminates and reports a valid winner.

> Stop the staircase where the pitch is met — not where ideas run out. New ideas
> during the build are tested against the NOT-list.

## 4. Acceptance tests (named before the code exists)

| Test | Asserts | Layer (verifiering.md) |
|---|---|---|
| `test_rules` | legal-move generation + flipping correct on hand-crafted positions (incl. multi-direction captures, edge/corner cases, forced-pass) | 1 |
| `test_integrity` | from the standard start, both sides always have a reachable path to a terminal state (no dead board) — the Othello analog of solvability | 2 |
| `test_fixtures` | **verify the verifier** (borrowed from game-builder §21.1): deliberately broken states — a move that flips nothing must be *rejected* as illegal; a full board must read `is_terminal`; a swapped direction table must make `test_rules` go RED | 1 |
| `test_bot` | a scripted bot plays a full game to a valid terminal (`score` sums to 64, one/zero winner) | 4 |
| `test_determinism` | same seed → identical bot-vs-bot game transcript | 1 |
| `test_flow` | turn alternation, auto-pass, win/draw banner, reset | 1 |

## 5. Feel rubric (derived from brief §2)

Fixed questions a fresh-context reviewer answers against a Fas-5 recording. Each
target-feel point → 1–2 observable questions:

1. Does a captured run **flip one disc at a time** (a visible cascade), not all at once?
2. Does each disc **read as a physical stone** — soft shadow, slight thickness, weight — rather than a flat token?
3. Is the flip **legible** — can you tell exactly which discs your move captured, from the animation alone?
4. Does the board read as a **warm, candlelit fantasy object** (per the vision brief), not a flat UI grid?
5. On placement, is there a **soft settle** (tiny bounce/clack), not a hard instant snap?

## 6. Definition of Done

- [ ] Every milestone's Testable line green.
- [ ] The bot plays a full game start → terminal → valid winner.
- [ ] Full match recorded; feel rubric answered with no un-accepted deviations.
- [ ] README = handbook (file map, run/test/see, staircase with (done) marks,
      automatable-vs-not).
- [ ] Graphics: board + both markers are pipeline-`validated`, delivered, and
      import-verified in Godot (the asset side of DoD — see `examples/othello/assets/`).
- [ ] User has confirmed against the pitch sentence.

### Note on this demo's scope

This build spec is **complete** (a real build would follow the whole staircase),
but the current deliverable stops after the **graphics** are produced and shown
assembled on a board (M2's asset side, rendered standalone) — the playable code
(M1/M3/M4) is specced, not built, this round. The asset requests, generators,
and a rendered fantasy Othello board live under `examples/othello/`.
